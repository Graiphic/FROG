<h2 id="overview">Overview</h2>

<p>
This repository defines the official specification of <strong>FROG — Free Open Graphical Language</strong>.
</p>

<p>
FROG is an open, hardware-agnostic graphical dataflow language designed to provide a durable and independent foundation for graphical programming.
</p>

<p>
The fundamental executable unit of the language is called a <strong>Frog</strong>.
A Frog is a self-contained graphical program represented as a structured JSON file (<code>.frog</code>).
</p>

<hr/>

<h2 id="what-a-frog-describes">What a <code>.frog</code> describes</h2>

<p>
A <code>.frog</code> file describes a complete Frog program:
</p>

<ul>
  <li><strong>Interface</strong>: the public typed inputs and outputs</li>
  <li><strong>Connector</strong>: how interface ports are positioned when the Frog is used as a node</li>
  <li><strong>Diagram</strong>: the executable dataflow graph (nodes, edges, layout)</li>
  <li><strong>Front panel</strong>: the interaction layer (controls, indicators, layout)</li>
  <li><strong>Metadata</strong>: identity, documentation, versioning, tags</li>
  <li><strong>Optional sections</strong>: icon, catalog, IDE preferences, IR cache</li>
</ul>

<p>
A <code>.frog</code> file MUST NOT contain hidden compiled binaries.
It is a transparent source representation.
</p>

<hr/>

<h2 id="frog-file-tree">FROG file tree</h2>

<p>
A Frog can exist in multiple forms, depending on the workflow.
</p>

<pre>
FROG
│
├─ Source (canonical)
│   └─ example.frog
│
├─ Optional build/cache artifacts
│   └─ example.frog.cache
│
└─ Optional distribution artifact (diagram removed)
    └─ example.frogbin
</pre>

<p>
The <code>.frog</code> file is the canonical source.
Cache and distribution artifacts are optional and can be generated by tools.
</p>

<hr/>

<h2 id="file-types-source-cache-distribution">File types: source, cache, distribution</h2>

<h3>1) <code>.frog</code> — Source (canonical)</h3>
<ul>
  <li>Human-readable JSON</li>
  <li>Version control friendly</li>
  <li>Contains diagram + interface + front panel</li>
  <li>May include optional sections (icon, catalog, IDE preferences)</li>
</ul>

<h3>2) <code>.frog.cache</code> — Cache (optional)</h3>
<ul>
  <li>Optional file generated by tools</li>
  <li>May contain a cached intermediate representation (IR)</li>
  <li>Used to accelerate loading/validation</li>
  <li>Safe to delete and regenerate</li>
</ul>

<h3>3) <code>.frogbin</code> — Distribution (optional)</h3>
<ul>
  <li>Optional artifact intended for distribution</li>
  <li>May omit the diagram entirely</li>
  <li>May contain IR and/or compiled target payloads</li>
  <li>Used when source disclosure is not desired</li>
</ul>

<hr/>

<h2 id="frog-json-top-level">FROG JSON top-level structure</h2>

<p>
A <code>.frog</code> file is a JSON object with the following recommended top-level structure:
</p>

<pre>
{
  "spec_version": "0.1",

  "metadata": {},
  "interface": {},
  "connector": {},

  "diagram": {},
  "front_panel": {},

  "icon": {},
  "catalog": {},
  "ide": {}
}
</pre>

<p>
Only a subset of fields are required. Optional sections MUST NOT change execution semantics.
</p>

<hr/>

<h2 id="sections">Sections</h2>

<h3 id="metadata">Metadata</h3>

<p>
Defines the identity and documentation of the Frog.
</p>

<pre>
"metadata": {
  "name": "Add",
  "description": "Adds two floating-point numbers.",
  "author": "Author Name",
  "program_version": "0.1.0",
  "tags": ["math", "primitive"]
}
</pre>

<hr/>

<h3 id="interface">Interface</h3>

<p>
Defines the public typed inputs and outputs of the Frog.
</p>

<pre>
"interface": {
  "inputs": [
    { "id": "a", "type": "float64" },
    { "id": "b", "type": "float64" }
  ],
  "outputs": [
    { "id": "result", "type": "float64" }
  ]
}
</pre>

<p>
Interface ports MUST be consistent with ports reachable in the diagram.
</p>

<hr/>

<h3 id="connector">Connector</h3>

<p>
Defines the graphical mapping of interface ports when a Frog is used as a node inside another diagram.
</p>

<pre>
"connector": {
  "pattern": "2x2",
  "ports": [
    { "interface_port": "a", "side": "left", "index": 0 },
    { "interface_port": "b", "side": "left", "index": 1 },
    { "interface_port": "result", "side": "right", "index": 0 }
  ]
}
</pre>

<p>
Each connector port MUST reference an existing <code>interface</code> port.
</p>

<hr/>

<h3 id="diagram">Diagram</h3>

<p>
Defines the executable dataflow graph.
</p>

<pre>
"diagram": {
  "nodes": [],
  "edges": []
}
</pre>

<p>
Nodes and edges MAY include layout data for editing and visualization.
Layout MUST NOT affect execution semantics.
</p>

<hr/>

<h3 id="front-panel">Front panel</h3>

<p>
Defines the interaction layer, including controls and indicators with positions and sizes.
</p>

<pre>
"front_panel": {
  "controls": [],
  "indicators": [],
  "layout": {}
}
</pre>

<p>
Front panel elements MUST bind explicitly to diagram ports.
</p>

<hr/>

<h3 id="icon">Icon</h3>

<p>
Optional 40×40 SVG icon used to represent the Frog when embedded as a node.
</p>

<pre>
"icon": {
  "size": 40,
  "svg": "&lt;svg&gt;...&lt;/svg&gt;"
}
</pre>

<hr/>

<h3 id="catalog">Catalog</h3>

<p>
Optional metadata used for indexing and discovery in development environments.
This section MUST NOT affect execution.
</p>

<pre>
"catalog": {
  "is_example": true,
  "title": "Basic Addition Example",
  "keywords": ["math", "example"],
  "categories": ["Examples", "Mathematics"]
}
</pre>

<hr/>

<h3 id="ide">IDE preferences</h3>

<p>
Optional tool preferences interpreted by development environments.
Runtimes MUST ignore this section.
</p>

<pre>
"ide": {
  "diagram": {
    "background_color": "#202020",
    "grid": true
  },
  "front_panel": {
    "background_color": "#ffffff"
  },
  "execution": {
    "debug_enabled": true,
    "reentrant_hint": false
  }
}
</pre>

<hr/>

<h3 id="cache">Cache (optional IR)</h3>

<p>
IR caching is supported through a separate file (<code>.frog.cache</code>) or an optional cache object in tooling workflows.
</p>

<p>
If a cache object exists, runtimes MUST treat it as optional and non-authoritative.
</p>

<pre>
"cache": {
  "ir_version": "0.1",
  "generated_by": "frog-ide",
  "generated_at": "2026-03-04T12:00:00Z",
  "ir": {}
}
</pre>

<hr/>

<h2 id="execution-model">Execution model</h2>

<p>
Execution follows a pure dataflow model:
</p>

<ul>
  <li>A node becomes eligible for execution when all required input data are available.</li>
  <li>Execution order MUST be derived solely from data dependencies.</li>
  <li>Nodes MAY execute in parallel if data dependencies allow.</li>
</ul>

<p>
A <code>.frog</code> file MUST NOT be executed directly.
It MUST first be validated and transformed into an intermediate representation prior to execution.
</p>

<hr/>

<h2 id="validation-rules">Validation rules</h2>

<ul>
  <li>Node identifiers MUST be unique within a diagram.</li>
  <li>Edge identifiers MUST be unique within a diagram.</li>
  <li>All edges MUST reference existing nodes and ports.</li>
  <li>Connected ports MUST be type-compatible.</li>
  <li>Connector ports MUST reference existing interface ports.</li>
  <li>Front panel bindings MUST reference existing diagram ports.</li>
</ul>

<hr/>

<h2 id="json-ordering">JSON ordering and canonical formatting</h2>

<p>
JSON object key order MUST NOT be considered significant.
Implementations MUST accept valid <code>.frog</code> files regardless of key ordering.
</p>

<p>
For readability and stable diffs, writers SHOULD emit keys in the recommended canonical order:
</p>

<pre>
spec_version
metadata
interface
connector
diagram
front_panel
icon
catalog
ide
</pre>

<hr/>

<h2 id="normative-terminology">Normative terminology</h2>

<ul>
  <li><strong>MUST</strong> — required for compliance</li>
  <li><strong>SHOULD</strong> — recommended</li>
  <li><strong>MAY</strong> — optional</li>
  <li><strong>MUST NOT</strong> — prohibited</li>
</ul>
